Introduccion a la base de Datos y los Datos

¿Qué son las Bases de Datos?
Imagina una base de datos como una gran biblioteca digital o un archivador muy organizado donde guardas información. Pero no es cualquier información; es información que está relacionada entre sí y que puedes buscar, actualizar o eliminar fácilmente.
Piénsalo así: si tuvieras que organizar todos los libros de una biblioteca al azar, sería imposible encontrar uno específico. Una base de datos hace el trabajo de organizar esa información de forma lógica para que siempre sepas dónde encontrar lo que necesitas. En resumen, una base de datos es una colección organizada de información que se almacena y se accede electrónicamente. Está diseñada para manejar grandes cantidades de datos de forma eficiente, permitiendo que múltiples usuarios y programas accedan a ellos al mismo tiempo.
¿Qué son los Datos?
Los datos son la materia prima de las bases de datos. Son hechos, cifras, estadísticas, textos, imágenes, audios o cualquier otra pieza de información sin procesar. Por sí solos, los datos pueden no tener mucho sentido.
Por ejemplo, el número "25" es un dato. Pero si decimos "25 años", ese dato ya tiene un contexto y se convierte en información útil. La información es el resultado de procesar, organizar y estructurar los datos para darles significado.
En una base de datos, los datos se organizan en unidades más pequeñas, como tablas, que a su vez contienen registros (filas) y campos (columnas). Cada campo contiene un tipo específico de dato (por ejemplo, nombres, edades, fechas).

Entendiendo las Bases de Datos

¿Cómo Entender las Bases de Datos?
Para entender mejor las bases de datos, piensa en cómo organizas la información en tu día a día.
Por ejemplo, una lista de contactos en tu teléfono o un catálogo de productos en una tienda. Las bases de datos nos permiten hacer varias cosas importantes:
• Almacenar: Guardar una gran cantidad de información de forma segura y duradera.
• Organizar: Estructurar la información de manera lógica para que sea fácil de encontrar y usar.
Imagina que cada tipo de dato tiene su "casillero" o campo específico. Múltiples casilleros
forman una tabla.
• Recuperar: Buscar y obtener rápidamente la información que necesitas. ¿Necesitas saber todos los clientes que viven en Girón? Una base de datos te lo puede decir en segundos.
• Modificar: Actualizar o cambiar la información existente. Si un cliente cambia su número de teléfono, es sencillo actualizar ese dato.
• Eliminar: Borrar información que ya no es necesaria.
Componentes clave:
• Tablas: Son el elemento principal de una base de datos. Imagínalas como hojas de cálculo donde cada columna es un campo (ej. Nombre, Edad) y cada fila es un registro (ej. "Juan Pérez", "30"). • Registros (Filas): Representan una única entrada de datos completa. Por ejemplo, todos los datos de un solo estudiante.
• Campos (Columnas): Son las categorías específicas de datos. Cada campo almacena un tipo particular de información (ej. solo nombres, solo fechas).

Representaciones Conceptuales de las bases de Datos

¿Cómo Hacer Representaciones Conceptuales de una Base de Datos?
Antes de construir una casa, un arquitecto hace planos. De la misma manera, antes de crear una base de datos, es fundamental hacer un diseño conceptual. Este diseño nos ayuda a entender qué información vamos a guardar y cómo se relaciona entre sí, sin preocuparnos aún por los detalles técnicos de cómo se va a implementar.
La representación conceptual más común es el Modelo Entidad-Relación (MER). Piensa en él como un mapa que muestra:
• Entidades: Son los "sustantivos" o cosas importantes sobre las que queremos guardar información. Pueden ser personas, objetos, lugares o eventos. Por ejemplo: Estudiante, Curso, Libro. En el MER, las entidades se representan usualmente con rectángulos.
• Atributos: Son las características o propiedades de las entidades. Son los "adjetivos" o detalles que describen una entidad. Por ejemplo, para la entidad Estudiante, los atributos podrían ser: nombre, apellido, edad, número de identificación. Los atributos se representan con óvalos conectados a sus entidades.
• Relaciones: Son las conexiones o asociaciones entre dos o más entidades. Indican cómo las entidades interactúan entre sí. Por ejemplo, un Estudiante "se inscribe en" un Curso. Las relaciones se representan con rombos conectando las entidades.
Ejemplo simple:
Imagina que quieres registrar la información de estudiantes y los cursos que toman.
• Entidad: Estudiante (atributos: ID_Estudiante, Nombre, Apellido)
• Entidad: Curso (atributos: ID_Curso, Nombre_Curso, Créditos)
• Relación: "se inscribe en" entre Estudiante y Curso. Un estudiante puede inscribirse en varios cursos, y un curso puede tener varios estudiantes.
El MER nos ayuda a visualizar la estructura de los datos, identificar qué información es relevante y asegurar que la base de datos pueda manejar las necesidades del negocio. Es el primer paso crucial en el diseño de cualquier base de datos sólida.

Cardinalidad en el modelo Entidad-Relación

Cardinalidad de las Relaciones en el MER
Además de saber que las entidades se relacionan, es crucial entender cuántas instancias de una entidad se relacionan con cuántas instancias de otra. Esto se conoce como cardinalidad de la relación. La cardinalidad nos dice cuántas ocurrencias de una entidad se asocian con cuántas ocurrencias de otra entidad.
Las cardinalidades más comunes son:
1. Uno a Uno (1:1): Cada ocurrencia de la entidad A se relaciona con una y solo una ocurrencia de la entidad B, y viceversa.
◦ Ejemplo: Un Ciudadano tiene un y solo un Número de Identificación, y un Número de Identificación pertenece a un y solo un Ciudadano.
2. Uno a Muchos (1:M): Una ocurrencia de la entidad A se relaciona con cero o muchas ocurrencias de la entidad B, pero cada ocurrencia de la entidad B se relaciona con una y solo una ocurrencia de la entidad A.
◦ Ejemplo: Un Departamento puede tener muchos Empleados, pero cada Empleado pertenece a un y solo un Departamento.
3. Muchos a Muchos (M:M): Una ocurrencia de la entidad A se relaciona con cero o muchas ocurrencias de la entidad B, y una ocurrencia de la entidad B se relaciona con cero o muchas ocurrencias de la entidad A.
◦ Ejemplo: Un Estudiante puede inscribirse en muchos Cursos, y un Curso puede tener muchos Estudiantes inscritos.
Entender la cardinalidad es vital porque afecta directamente cómo se diseñan las tablas en la base de datos real (el modelo lógico) y cómo se almacenan y recuperan los datos de manera eficiente.

Claves primarias y foráneas

Claves Primarias y Foráneas
Una vez que tenemos nuestro diseño conceptual (MER) con entidades, atributos y relaciones, necesitamos una forma de implementar esas relaciones en un sistema de base de datos real. Aquí es donde entran las claves.
1. Clave Primaria (Primary Key - PK):
◦ Es uno o un conjunto de atributos que identifica de forma única cada registro (fila) en una tabla.
◦ Es como el DNI o número de identificación de cada fila: no puede haber dos filas con la misma clave primaria y no puede ser nula (vacía).
◦ Ejemplo: En una tabla de Estudiantes, el atributo ID_Estudiante podría ser la clave primaria. 2. Clave Foránea (Foreign Key - FK):
◦ Es uno o un conjunto de atributos en una tabla que hace referencia a la clave primaria de otra tabla.
◦ Sirve para establecer y mantener la relación entre dos tablas.
◦ Ejemplo: En una tabla de Cursos que tienen Estudiantes inscritos, podríamos tener una clave foránea ID_Estudiante que apunte al ID_Estudiante de la tabla de Estudiantes. Esto nos permite saber qué estudiante está en qué curso.
Las claves primarias y foráneas son fundamentales para la integridad de los datos (asegurar que los datos sean precisos y consistentes) y para poder realizar búsquedas y uniones entre diferentes tablas de forma eficiente. Son el pilar de las bases de datos relacionales, que son las más comunes hoy en día.
